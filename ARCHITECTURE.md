## 1. What This Service Does

`my-claude-code-board` is not a deployed application — it is a **Claude Code developer infrastructure board**: a central repository of session protocols, Claude hooks, brand assets, and shared knowledge that governs how Claude Code operates across every HRI project. When a developer opens Claude Code in any HRI repo, the session-start hook in that repo fetches protocols from this board (or from `hri-template-repository`). The board standardizes four things: (1) how every session starts (context hygiene, reading project state, confirming scope), (2) how every session ends (updating CLAUDE.md project knowledge, updating `hri-stack-learnings.md`, writing to the HRI Systems Registry, and updating the Project Status Tracker), (3) guardrails on git push operations (push-guard hook blocks accidental cross-owner pushes), and (4) brand identity assets and the `hope-rises-brand` skill so Claude can produce on-brand deliverables in any session. The board also holds setup documentation for the computer-use MCP server. There is no scheduled job, no Cloud Run service, and no data pipeline — this repo's value is entirely in the protocols and assets it provides to other sessions.

## 2. How It Runs

| Attribute | Value |
|-----------|-------|
| Trigger | Invoked on-demand — fetched by Claude Code sessions in other HRI repos at session start/end |
| Schedule | None — no cron, no Cloud Scheduler, no deployment |
| Runtime environment | Local developer machine (macOS); the `session-start.sh` hook runs in a remote Claude Code environment only when `CLAUDE_CODE_REMOTE=true` |
| Expected duration | The hooks execute in seconds; the protocols they reference are read and followed over the course of a full work session |
| Deployed service | None |

The `session-start.sh` hook installs `clasp` (Google Apps Script CLI) if missing, and writes a global `~/.claude/CLAUDE.md` containing the HRI Systems Registry endpoint and payload templates. This only runs in remote (cloud-hosted) Claude Code environments — it exits immediately on local machines.

## 7. What Breaks If This Stops Running

This repo does not run continuously — it is a reference and protocol store. However, if it became unavailable or was corrupted:

- **Session discipline collapses across all HRI projects.** Every HRI Claude Code session is supposed to read session-start and session-end protocols. If those protocols are missing or stale, sessions will skip context hygiene, skip Registry updates, and skip Project Status Tracker updates. Operational visibility degrades immediately.
- **The HRI Systems Registry goes stale.** The Registry (Sheet ID: `115R3nf55qL23dftoZQg3jOQh5Drqvj-Cd08lsyR_Z2g`) is the central dashboard for all deployed systems, credentials, and build history. Without protocol enforcement, it will drift from reality within days.
- **Brand consistency breaks.** The `skills/hope-rises-brand` skill — including logo files, color palette, font specs, and the `letterhead_template.docx` and `presentation_template.pptx` — would be unavailable. Claude-generated deliverables would lose brand consistency.
- **Push-guard protection is lost.** The `push-guard.sh` hook would not fire, removing the cross-owner push warning that prevents accidental force-pushes to wrong repositories.
- **New team members cannot set up computer use.** The `hri-computer-use-setup.md` is the only setup guide for the computer-use MCP server.

## 8. Three Most Likely Failure Modes

### Failure Mode 1: Sheets Bridge API rejects session-end Registry updates

**What goes wrong:** The Apps Script bridge endpoint returns `"action": "error"` because a column name in the payload doesn't match the actual sheet headers exactly (case-sensitive). This happens when the Registry sheet is restructured or a tab is renamed, but the protocol templates in `session-start.sh` and the protocol markdown files are not updated.

**Symptom:** The `curl` two-step POST completes without a network error, but the JSON response body contains `"action": "error"` with a message like `"Column 'Notes' not found"`. The Registry row is not written.

**How to fix:**
1. Open the HRI Systems Registry (`115R3nf55qL23dftoZQg3jOQh5Drqvj-Cd08lsyR_Z2g`) and check the exact header row for the affected tab.
2. Update the column name in the payload to match exactly (case-sensitive).
3. Retry the two-step curl from `session-end-protocol.md`.
4. If the headers were changed at the sheet level, update `session-start.sh` and `session-end-protocol.md` in this repo and commit.

---

### Failure Mode 2: ADC credentials expired or missing required scopes — Project Status Tracker update fails

**What goes wrong:** The Python snippet in the session-end protocol calls the Sheets API v4 directly using ADC credentials. If credentials are expired, have the wrong scopes (`drive.readonly` instead of `drive` full), or were generated with gcloud's default OAuth client (which Google blocks for the `drive` scope), the Sheets API returns a 403 or token refresh error.

**Symptom:** The Python script exits with `google.auth.exceptions.RefreshError`, `403 Forbidden`, or `HttpError 403` when calling `service.spreadsheets().values().get(...)`. The Project Status Tracker row is not updated.

**How to fix:**
1. Re-authenticate using the HRI custom OAuth client file (stored in the `claude-repo` shared Drive folder as `hri-oauth-client.json`):
   ```bash
   gcloud auth application-default login \
     --client-id-file="$HOME/Downloads/hri-oauth-client.json" \
     --scopes="https://www.googleapis.com/auth/cloud-platform,https://www.googleapis.com/auth/drive"
   ```
2. Confirm credentials are at `~/.config/gcloud/application_default_credentials.json`.
3. Re-run the Python session-end tracker update.
4. Do NOT use `gcloud auth application-default login` without the custom client file — this is the most common cause of 403 errors on Drive-scoped calls.

---

### Failure Mode 3: Protocol files become out of sync across repos

**What goes wrong:** The session-start and session-end protocols are designed to be fetched fresh from `hri-template-repository` at runtime. However, local copies of `session-start-protocol.md`, `session-end-protocol.md`, and `hri-stack-learnings.md` exist in this repo. If the canonical copies in `hri-template-repository` are updated but the local copies here are not, sessions that read the local files instead of fetching from GitHub will follow stale instructions.

**Symptom:** A session follows an outdated protocol step — for example, missing a new required Registry tab, using a wrong column header, or missing a new step (like Step 4, the Project Status Tracker update, which was added later). The symptom is often silent: the session appears to complete normally but a registry or tracker row is missing.

**How to fix:**
1. Compare this repo's local protocol files against the canonical versions in `hri-template-repository` via:
   ```bash
   gh api /repos/Hope-Rises-International/hri-template-repository/contents/session-end-protocol.md \
     --jq '.content' | base64 -d > /tmp/canonical-session-end.md
   diff /Users/ccalvert/claude-work/my-claude-code-board/session-end-protocol.md /tmp/canonical-session-end.md
   ```
2. If the local copies are stale, overwrite them with the canonical versions and commit.
3. As a rule: always fetch from `hri-template-repository` at session start rather than relying on local copies. The `CLAUDE.md` instructions do this correctly; verify sessions are following those instructions.

## 9. How to Manually Re-Run

This repo has no automated job to re-run. The "manual run" scenarios are:

### Re-run the HRI Systems Registry update (Sheets Bridge)

Replace placeholder values with actuals, then run:

```bash
PAYLOAD=$(cat <<'EOF'
{
  "apiKey": "b3317d76-a5f3-4657-9881-6f087455173d",
  "sheetId": "115R3nf55qL23dftoZQg3jOQh5Drqvj-Cd08lsyR_Z2g",
  "operations": [
    {
      "tab": "Repositories",
      "mode": "upsert",
      "matchColumns": ["Repository"],
      "data": {
        "Repository": "my-claude-code-board",
        "Platform": "GitHub",
        "URL": "https://github.com/bsimmons3rd/my-claude-code-board",
        "Status": "Active",
        "Last Updated": "YYYY-MM-DD",
        "Notes": "BRIEF_DESCRIPTION"
      }
    },
    {
      "tab": "Build Log",
      "mode": "append",
      "data": {
        "Date": "YYYY-MM-DD",
        "Builder": "Claude Code",
        "System": "my-claude-code-board",
        "Action": "WHAT_WAS_DONE",
        "Issues Encountered": "None",
        "Resolution": "N/A",
        "Time Spent": "ESTIMATE",
        "Notes": "CONTEXT"
      }
    }
  ]
}
EOF
)

REDIRECT_URL=$(curl -s -o /dev/null -w "%{redirect_url}" -X POST \
  "https://script.google.com/macros/s/AKfycbzen_jFFDULJQhj7rPK8khEopSRR1YIusTDqI7dpQPrefdi7Lxcv_J5UimlolVg49i-vQ/exec" \
  -H "Content-Type: application/json" \
  -d "$PAYLOAD") && curl -s -L "$REDIRECT_URL"
```

Check the response — `"action": "error"` means a column name didn't match. `"action": "upsert"` or `"action": "append"` means success.

### Re-run the Project Status Tracker update

1. Ensure ADC credentials are valid (see Failure Mode 2 above for re-auth command).
2. Open a Python session and run the full Python block from `session-end-protocol.md`, setting `REPO_NAME = "my-claude-code-board"` and `SESSION_DAYS` to the appropriate value.
3. The script will find the row in `Active!A:N` of Sheet ID `1KD7Zds9WlIQRubD2z7r2SmcTIWbIQCQtkfE713P88Ng`, recalculate LOC/hours/$ values, and update columns B–N.

gh api /repos/Hope-Rises-International/hri-template-repository/contents/session-end-protocol.md \

# Compare against local copies
diff /Users/ccalvert/claude-work/my-claude-code-board/session-start-protocol.md /tmp/session-start-protocol.md
diff /Users/ccalvert/claude-work/my-claude-code-board/session-end-protocol.md /tmp/session-end-protocol.md
